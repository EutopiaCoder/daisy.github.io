redis字典学习笔记

  • redis的hash怎么实现的？
  • rehash过程讲一下？和java的ConcurrentHashMap的rehash有什么区别？

大道理都懂，问细节就崩溃，《Redis设计与实现》这本书不错，仔细读了一下字典这一节，记一下笔记。（还能蹭点C语言）

redis数据结构:

字典底层由哈希表实现

哈希表:
dict.h/dictht

哈希表节点

dict.h/dictht/dictEntry

字典

dict.h/dict

redis在添加新的键值对时，会计算哈希值和索引值，然后再根据索引值插入。

hash = dict -> type -> hashFunction(key);

index = hash & dict->hx[x].sizemask

hashFunction(key)使用的算法是Murmurhash2

哈希冲突：

类似于java中的ConcurrentHashMap，采用拉链法，新值永远放在链表头部

rehash步骤：

  1. 为ht[1]分配空间，大小取决于ht[0].used的值

  2. 将保存在ht[0]中的所有键值对rehash到ht[1]上

  3. ht[0]包含的所有键值对迁移完之后释放ht[0]，将ht[1]设置为ht[0]，并在ht[1]创建一个空白哈希表，为下次rehash做准备

redis是单线程，数据量过大会导致服务一段时间内停止服务，所以采用的是渐进式rehash

渐进式rehash步骤：

1. 为ht[1]分配空间，大小取决于ht[0].used的值
 2. 字段中维持一个索引计数器变量rehashidx，并将它的值设置为0，表示rehash工作正式开始
 3. rehash期间，每次对字典的crud都会将ht[0]的键值对rehash到ht[1]，工作完成后rehashidx++
 4. 全部完成之后rehashidx = -1，表示rehash操作完成

在这期间所有的crud都会在2个哈希表上进行，先ht[0]，没找到就ht[1]，新增操作会一律保存在ht[1]上，保证ht[0]的size只减不增，直至变成空表

​

回到最初的问题，和java的ConcurrentHashMap有何区别

redis是单线程渐进式

ConcurrentHashMap是多线程协同扩容

扩容速度：平均来说ConcurrentHashMap更快

读：没啥区别

写：ConcurrentHashMap可能会去帮着扩容，平均来说redis更快

删：没啥区别

eureka实现步骤

之前记录feign实现步骤的时候，IOC容器实例化bean会调用FeignClientFactoryBean的getObject方法，会根据配置文件里的application.name拼装一个url，比如http://nagato，封装成InvocationHandler生成feign代理类。

后面在调用executeWithLoadBalancer的command.submit方法时，只分析到ribbon会调用EurekaClient根据application.name获取eureka的服务实例列表，再通过ribbon自己的算法得到对应的实例地址，紧接着调用subimt内部的ServerOperation类实现Http调用。

今天来看下eureka是怎么进行服务注册、续约、获取注册列表、服务下线和服务清除。

现在看源码还是不太熟练，不借助官方文档只能靠猜了，先启动个eureka注册中心吧，代码特别简单。

EurekaClient的子类DiscoveryClient有个方法叫register，我们看下注释

通过一个正确的RESTful请求注册一个eureka服务，应该就是你了，打个断点，我们再启动个服务注册上去

果然进断点了

往上找，发现是在InstanceInfoReplicator类里调用的，这块代码比较上，贴一下调用链。

1.DiscoveryClient初始化的时候会new一堆线程池，用来负责上面提到的服务注册、服务续约等功能

2.有服务要注册的时候statusChangeListener会监听到请求，调用instanceInfoReplicator.onDemandUpdate();

3.onDemandUpdate方法调用InstanceInfoReplicator.this.run();

4.InstanceInfoReplicator实现了runnable，调用run内部的discoveryClient.register()方法，即上面贴的register方法。顺带一提这个方法的finally块也会同时开启服务续约的定时任务。

调用完register方法之后发现注册中心的控制台输出了nagato服务注册的log，代表EurekaServer接收到了请求并注册到了eureka上，猜了半天名字猜到了EurekaBootStrap这个类。

这个方法没有register方法，查询init开头找到了initEurekaServerContext这个方法

猫腻应该就在PeerAwareInstanceRegistryImpl里面了，进去看下，果然有register方法在这里

父类的register方法太长，把注释贴一下

和查EurekaClient一样，我们往上查哪里调用的，最终发现在ApplicationResource方法暴露了一个接口

为了保证没找错，这里也打一个断点，重启一次nagato服务，发现先进了eureka client那边，然后调用了eureka server这边，没毛病！服务成功注册到了eureka上。

我们看一下com.netflix.eureka.resouces，ApplicationResource就在这个文件夹里

可以类比理解为平时开发的那个Controller层了，像什么续约，获取注册列表，清除服务都在这里。

有一次发现注册中心警告了，也和服务清除有关系，百度一段粘过来先

自我保护模式：Eureka Server接收的续约低于配置的85%时，开启自我保护模式，不再清除服务。

feign实现步骤

记录一下feign的实现原理，直接把流程干货提炼出来方便以后回忆。

启动部分

初始化beanDefinite

1.启动springboot初始化IOC容器，调用SpringApplication.refreshContext;

2.refresh()的invokeBeanFactoryPostProcessors，实例化所有注册了BeanFactoryPostProcessor的bean;

3.invokeBeanDefinitionRegistryPostProcessors，遍历所有currentRegistryProcessors，执行postProcessBeanDefinitionRegistry方法。

4.因为@EnableFeignClients注入了FeignClientsRegistrar，FeignClientsRegistrar实现了ImportBeanDefinitionRegistrar，到这里会执行registerBeanDefinitions。

5.registerBeanDefinitions包含2个方法，registerDefaultConfiguration和registerFeignClients。

6.registerDefaultConfiguration先再判断一次启动类是否有@EnableFeignClients，然后注册FeignClientSpecification。

7.registerFeignClients用于扫描所有使用了@FeignClient的类，将注解的信息全部取出，通过BeanDefinitionBuilder组合为BeanDefinition，注入到IOC容器中

实例化bean

1.refresh()的finishBeanFactoryInitialization(beanFactory)，实例化bean。

（我自己写了个DemoController，@Autowired了一个@FeignClient修饰的DemoService，调用链是先实例化DemoController，发现要注入DemoService调用了DepencyDescriptor.resloveCandidate方法，至于顺序后续再研究下）

2.DemoService的beanFactory是FeignClientBeanFactory，会调用其getObject方法。

3.先拼接url，如服务端的application.name为nagato，url为http://nagato。

4.调用loadBalance方法

5.若有hystrix熔断器逻辑则进行配置。

6.最后到ReflectiveFeign调用newInstance并返回到容器中，其实就是jdk的动态代理。

调用部分

1.调用DemoService代理类后进入invoke方法，SynchronousMethodHandler的invoke方法中首先create一个HTTP模板RequestTemplate(GET /jwt/hello HTTP/1.1)。

2.克隆一个Retryer重试Bean，进入executeAndDecode方法。

3.之前拼接了一个url，再次拼接为(GET http://nagato/jwt/hello HTTP/1.1)

4.调用client发送请求并解析返回结果，这里的client非常关键。

（查看FeignAutoConfiguration可以发现@ConditionalOnClass，如果pom里有feign-httpclient，则使用httpclient调用，如果pom里有feign-okhttp，则使用okhttp调用，我2个都没导入，这里用的client是LoadBalancerFeignClient）

5.LoadBalancerFeignClient的executeWithLoadBalancer方法，分为两个步骤，command.submit实现负载均衡，subimt传参为ServerOperation匿名类，内部实现了call，http调用在这里实现。

6.call往里研究，跟踪到convertAndSend这一步，本质还是HttpURLConnection(我这次调用的url为http://join-PC:9001/jwt/hello)。

7.刻在DNA里的out.write(request.body())

7.得到返回的报文，decode处理后作为结果返回。

8.进入InvocableHandlerMethod的invoke方法，回到原来调用DemoService方法，继续执行。
